# Dashboard Vision: From Data to Insight

## The Problem

The dashboard currently shows data. Data is not insight. A chart of
bill shock events tells you what happened. It does not tell you what
it means, what to do about it, or whether things are getting better.

Rich needs insight, not data. The dashboard should tell him what the
business is doing, whether it's working, and what needs attention —
without requiring him to interpret raw numbers.

## The Insight Hierarchy

Build toward four levels, in order:

### Level 1 — Data (partially exists)
What happened. Charts, tables, figures per run. The dashboard has
some of this. Continue adding data as new modules are built. But
data alone is not enough.

### Level 2 — So What (build next)
What does this data mean? After every run, for each major data area,
generate a "so what" interpretation:

- **Trading**: "The hedge book performed well in 2023 — actual margin
  was £47k above the naked counterfactual. The adaptive EWMA pricer
  responded to rising prices 3 weeks earlier than the previous model."

- **Customers**: "Bill shock events increased 40% in 2022. 6 customers
  experienced shocks >150%. Retention offers were blocked in 4 cases
  by the margin guard — this is appropriate given crisis-year margins,
  but means 4 high-risk customers left without an offer."

- **Risk**: "VaR breached the stressed floor 3 times in Q4 2021.
  The risk committee responded correctly but hedge fractions were
  already near the 85% mandate floor, limiting room to manoeuvre."

- **Financial**: "Net margin is 4.4% of revenue — within the 2-5%
  industry benchmark. However, I&C customers account for 94% of total
  net margin. Without C_IC1 and C_IC2 the book would be loss-making."

The "so what" is generated by the agent after each run using Qwen
for local inference — no frontier tokens spent on routine reporting.
It appears in the dashboard as a narrative card per section, and is
also sent as a NTFY digest to Rich.

### Level 3 — Coherence (build after Level 2)
What do the findings mean across areas? Identify patterns that span
data categories:

"The 2022 data tells a single story: the company entered the crisis
under-hedged relative to its I&C exposure, bill shocks drove churn
in the residential book, and the margin guard correctly blocked
retention offers that would have deepened losses. The business
survived but the I&C book cross-subsidised the residential book
throughout the crisis. Without C_IC1 the company would have failed
in 2022."

Coherence insights appear as a top-level "Executive Summary" section
generated after every run. One paragraph. Direct. No hedging.

### Level 4 — Trend (build after Level 3)
Is the company getting better or worse? Compare current run against
previous runs across key metrics:

- Is the churn model getting more accurate? (company_est vs sim_churn_p gap)
- Is the retention strategy improving? (ROI trend across runs)
- Is the forward curve pricer getting better? (tariff error by year)
- Is margin improving or deteriorating?

Trend analysis requires run history. Store key metrics per run in
docs/observability/run_history.json. The dashboard trend charts read
from this file.

## The "So What" as a Prompt to Act

The most important property of Level 2-3 insights: they should prompt
action. Not just observation.

After generating the "so what", the agent should ask itself:
"What should the company do differently next run given these findings?"

This becomes a proposed action in the dashboard — visible to Rich as
a recommendation, and potentially auto-staged as a simulation parameter
change or phase proposal.

Examples:
- "Retention budget insufficient for crisis years — recommend relaxing
  margin guard threshold for customers with >5yr tenure."
- "I&C concentration risk: 94% of margin from 2 customers — recommend
  acquisition target of 2 new I&C customers in next run."
- "Churn model significantly underestimates crisis-year risk
  (company_est avg 0.06 vs sim_churn_p avg 0.31 in 2021) —
  recommend Phase X: crisis-regime churn model."

Recommendations that Rich acts on inform phase priorities. This is
the feedback loop that makes the dashboard a strategic tool, not a
reporting tool.

## Dashboard Functionality Requirements

The dashboard currently shows data passively. It needs to do things.

### Must have (build now alongside insight layer):
- **Filter by year**: see any single year's data across all sections
- **Filter by customer/segment**: see one customer's full history
- **Compare years**: 2021 vs 2019 side by side for any metric
- **Drill down**: click a bill shock event, see the customer, the
  tariff, what happened at next renewal
- **Anomaly highlighting**: automatically flag anything outside
  expected range (red/amber/green per metric vs benchmark)

### Should have (build after must-haves):
- **Export**: download any chart as PNG or data as CSV
- **Run comparison**: compare current run vs previous run
- **Timeline scrubber**: move through 2016-2025 and watch metrics change

### The "Talk to the Database" Interface (build after Level 2 insights):
A natural language query interface on the dashboard. Rich types a
question, the system answers using the simulation data.

Examples:
- "Which customers were unprofitable in 2022?"
- "Show me all retention offers where the customer churned anyway"
- "What was the forward curve error in crisis years?"
- "Which segment has the highest bill shock rate?"

Implementation: Claude API (claude-sonnet) with the run output JSON
as context. The agent pre-processes each run's data into a queryable
format. The dashboard sends the question + data context to the API
and displays the answer.

This is achievable now. It turns the dashboard from a report into
a thinking tool.

## Dashboard as a Standing Responsibility

### Rule: every phase includes a dashboard requirement.
When a new module is built, the phase is not complete until:
1. The new data is visible in the relevant dashboard section
2. A "so what" template exists for that data area
3. The insight generator knows how to interpret it

The epistemic verifier checks the SIM/company barrier at phase close.
Add an equivalent check: "is this new capability visible and
interpretable in the dashboard?"

### After every full sim run, the agent must:
1. Regenerate all dashboard data
2. Generate "so what" narratives for each section
3. Generate coherence executive summary
4. Compare against previous run (trend)
5. Produce recommended actions
6. Publish to poesys.net
7. Send NTFY digest with executive summary and top 3 recommendations
8. Store run metrics in docs/observability/run_history.json

### Monthly dashboard review (standing item):
Once a month, the agent produces a dashboard capability assessment:
- What data areas have no "so what" interpretation?
- What questions can't be answered from current data?
- What functionality is missing vs the vision above?
- What is the proposed improvement backlog?

This assessment goes to poesys.net/dashboard-review and is NTFY'd
to Rich. It drives dashboard development priorities.

## Implementation Sequence

**Phase A — So What generation:**
Build the insight generator for the 5 core areas (trading, customers,
risk, financial, operations). Qwen-powered, runs after every full sim
run. Output appears as narrative cards in the dashboard and as NTFY
digest.

**Phase B — Dashboard functionality:**
Filter by year, filter by customer, compare years, drill-down on
events, anomaly highlighting. These are pure frontend changes to
site/index.html — no simulation changes needed.

**Phase C — Coherence and trend:**
Executive summary generation. Run history storage. Trend charts.

**Phase D — Talk to the database:**
Natural language query interface using Claude API with run data as
context. Appears as a chat panel on the dashboard.

**Phase E — Standing process:**
Monthly dashboard capability assessment. Dashboard requirement in
every phase checklist.

## What Rich Gets

After this is built, after every sim run Rich gets:
- A NTFY with the executive summary and top 3 recommendations
- A dashboard he can explore by year, customer, segment
- A "so what" for every data area that tells him what it means
- A coherence narrative that tells the business story
- A chat interface to ask any question about the data
- A monthly assessment of what the dashboard can't yet do

This is the MI suite of a real company. It informs his steering.
It makes the simulation legible. And it drives the agent's own
priorities — because the "so what" identifies what needs fixing.
